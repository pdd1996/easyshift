| 科室   | 医院内最小排班单元，v1 仅支持单个科室              |
| 排班周期 | 以「周」为单位的管理区间，默认周一 00:00 至周日 23:59 |

| 绑定码  | 管理员为员工生成的一次性小程序绑定凭证               |
| 管理员  | 负责排班、发布、员工管理的角色，通常为护士长或科室排班员      |
| 员工   | 科室一线人员，通过小程序查看个人班表                |

## 2. 产品概述

### 2.1 背景

医院科室排班目前多依赖 Excel、纸质或微信群通知，存在以下问题：

- 排班耗时长，修改后通知不及时
- 员工查看班表不便，容易遗漏变更
- 大夜班、休息等规则靠人工记忆，易出错
- 变更缺少记录，事后难以追溯

### 2.2 产品定位

EasyShift 是一款面向**医院小型科室**的轻量排班工具：

- **Web 管理端**：供管理员完成排班、发布与基础管理
- **微信小程序**：供员工查看已发布的个人班表

v1 聚焦「排得出来、发得出去、看得到」，不追求全自动智能排班。

### 2.3 产品目标（v1）


| 目标  | 说明                           |
| --- | ---------------------------- |
| G1  | 管理员可在 15 分钟内完成一个 10 人科室的下周排班 |
| G2  | 班表发布后，员工可在小程序内查看本人班次，无需翻群消息  |
| G3  | 管理员可看到每天各班次人数是否达标，避免漏排关键班次   |
| G4  | 基础排班冲突可被系统提示，降低明显错误          |
| G5  | 排班变更有发布快照与操作记录，可追溯           |


### 2.4 非目标（v1 明确不做）

- 多院区、多科室、多级组织架构
- 一键自动排班 / AI 优化
- 复杂资质约束（如「每班必须有护师」）
- 与 HIS、考勤机、薪资系统对接
- 工时薪资、夜班费自动计算
- 员工端换班、请假（v1.5 规划，v1 仅预留数据模型）
- 独立 App（仅微信小程序）

## 3. 用户与场景

### 3.1 用户角色


| 角色    | 使用端   | 描述                 |
| ----- | ----- | ------------------ |
| 科室管理员 | Web   | 维护员工与班次类型，编辑并发布排班表 |
| 科室员工  | 微信小程序 | 绑定身份后查看已发布个人班表     |


v1 每个科室仅 **1 个管理员账号**，不支持多管理员协作。

### 3.2 用户画像

**管理员（护士长）**

- 每周固定时间排下周班表
- 熟悉 Excel，但对复杂系统耐心有限
- 关注：排班效率、大夜班公平、变更通知是否到位

**员工（护士 / 医技）**

- 主要用手机查看班表
- 希望打开即见，不想反复登录复杂系统
- 关注：我的班准不准、有没有被悄悄改掉

**字段规则**

- 工号：科室内唯一，2–20 字符

| 需求 ID        | 描述                     | 优先级 |
| ------------ | ---------------------- | --- |
| WEB-SHIFT-01 | 班次类型列表：名称、代码、时间段、颜色、排序 | P0  |
| WEB-SHIFT-02 | 新增 / 编辑 / 停用班次类型       | P0  |

| WEB-SHIFT-04 | 「休息」类班次可不填时间段          | P0  |
| WEB-SHIFT-05 | 可配置每个班次每日最低覆盖人数，作为排班表统计依据 | P0  |


**默认班次（可配置）**


| 代码  | 名称  | 默认时间        | 说明              |
| --- | --- | ----------- | --------------- |
| D   | 白班  | 08:00–16:00 | 日班              |
| E   | 小夜班 | 16:00–00:00 | 跨日，以开始时刻存储      |
| N   | 大夜班 | 20:00–08:00 | 跨日，以开始时刻 + 时长存储 |
| OFF | 休息  | —           | 显式休息            |
| SB  | 备班  | —           | 待命，v1 不单独校验规则   |


**字段规则**

- 代码：科室内唯一，1–10 字符
- 开始时间 + 时长（分钟）定义班次；跨日通过时长 > 当日剩余时间表达
- 颜色：用于排班表单元格展示
- 最低覆盖人数：非负整数，默认 0；如白班最低 3 人、大夜班最低 1 人
- 已被历史排班或发布快照引用的班次不可物理删除，只能停用；停用后不可新排班，历史展示不受影响

| ---------- | ----------------------------------------- | --- |
| WEB-SCH-01 | 周视图表格：行 = 员工，列 = 周一至周日                    | P0  |
| WEB-SCH-02 | 点击单元格选择班次类型，支持清空                          | P0  |
| WEB-SCH-03 | 排班表顶部展示当前周期、状态（草稿 / 已发布）                  | P0  |

- 同一员工、同一日期仅允许 1 条排班记录

### 4.3 跨端业务规则

#### 4.3.1 发布与可见性

```
管理员 Web 编辑草稿排班
       ↓
管理员点击发布 → 生成发布快照 version N
       ↓
小程序仅读取最新发布快照
       ↓
管理员继续修改 → 标记有未发布变更，但不影响小程序可见快照
       ↓
再次发布 → 生成发布快照 version N+1
```


| 规则   | 说明                                                                             |
| ---- | ------------------------------------------------------------------------------ |
| R-01 | 新建周期默认为 draft                                                                  |
| R-02 | 员工端不可见草稿排班，只读取最新发布快照                                                        |
| R-03 | 已发布周期若被修改，周期标记为「有未发布变更」，员工端继续展示上一版发布快照                            |
| R-04 | 再次发布后生成新版本快照，员工端读取最新版本                                                       |
| R-05 | 发布快照需冗余保存员工姓名、工号、班次名称、代码、时间段、颜色，避免历史展示受基础配置修改影响                  |


**推荐方案（v1）**：`schedule_periods` 保存管理员当前编辑态和未发布变更标记，员工端只读取 `schedule_publish_snapshots` 中最新版本。管理员编辑已发布周期时不回退员工端可见数据，直到再次发布。

#### 4.3.2 时间规则


| 规则   | 说明                                                                     |
| ---- | ---------------------------------------------------------------------- |
| T-01 | 排班日期以「班次开始所在日历日」为准                                                     |
| T-02 | 跨日班次（如 20:00–08:00）存储为 `work_date` + `start_time` + `duration_minutes` |
| T-03 | v1 不支持多时区，默认服务器东八区                                                     |
| T-04 | 周期按自然周，周一为一周起始日                                                        |


#### 4.3.3 权限


| 规则   | 说明                         |
| ---- | -------------------------- |
| P-01 | Web 端接口仅 `admin` 角色可访问管理功能 |
| P-02 | 小程序接口仅 `staff` 角色可访问       |
| P-03 | 员工只能读取**本人**排班数据           |
| P-04 | 管理员不可通过小程序端登录（v1）          |

#### 4.3.4 Token 生命周期

| 规则 | 说明 |
|------|------|
| A-01 | Web 管理端会话默认 7 天有效，由服务端通过 HttpOnly Cookie 下发与续期 |
| A-02 | 小程序 Bearer Token 默认 30 天有效，过期后需重新微信登录获取 |
| A-03 | 员工停用、解绑、重新绑定后，服务端必须拒绝该员工此前签发的旧 Token |
| A-04 | 管理员账号停用或密码重置后，服务端必须拒绝旧 Web 会话 |

## 5. 非功能需求

### 5.1 性能


| 指标        | 目标                       |
| --------- | ------------------------ |
| Web 排班表加载 | 10 人 × 7 天，首屏 < 2s（正常网络） |
| 小程序班表加载   | < 1.5s                   |
| 排班单元格保存   | 单次操作响应 < 500ms           |


### 5.2 可用性

- Web 支持 Chrome、Edge 最新两个大版本
- 小程序基础库版本按微信官方建议最低版本
- 关键操作有明确成功 / 失败反馈

### 5.3 安全

- 管理员密码加密存储（bcrypt 或同等方案）
- 全站 HTTPS
- API 鉴权：Web 管理端使用 HttpOnly Cookie 承载会话/JWT；小程序端使用 Bearer Token
- Web 端禁止将访问令牌存入 localStorage
- Web Cookie 需设置 `HttpOnly`、`Secure`、`SameSite=Lax` 或更严格策略；涉及写操作的接口需校验 `Origin` / `Referer`，作为 v1 CSRF 防护方案
- 小程序 Bearer Token 默认有效期 30 天；员工停用、解绑、重新绑定后，服务端必须拒绝旧 Token
- 小程序 `openid` 不在客户端暴露给其他用户
- 敏感操作（发布、解绑）有服务端权限校验

### 5.4 可靠性

- 排班数据写操作需事务保证一致性
- 发布操作建议具备「发布快照」或版本号，防止编辑中途员工看到不完整数据
- 发布班表必须在数据库事务内完成：锁定目标 `schedule_periods` 行、计算下一版本号、写入发布快照、更新周期发布信息、写入操作日志必须同事务提交
- 同一排班周期的并发发布请求必须串行化处理，避免生成重复版本号或员工端读取到不完整快照

### 5.5 合规与隐私

- 隐私政策说明收集的信息：微信 openid、工号、姓名、排班数据
- 员工数据仅本科室管理员可管理
- v1 不做公开分享链接

## 7. 数据模型（概要）

### 7.1 实体关系

```
departments 1 ── * employees
departments 1 ── * shift_types
departments 1 ── * schedule_periods

schedule_periods 1 ── * schedule_publish_snapshots

employees 1 ── * employee_binding_codes
```

### 7.1.1 各表含义（设计备注）

下表说明每张表在系统中的职责，以及与其他表的分工。实现 Drizzle schema 时可作为表注释依据。

| 表名 | 一句话含义 | 谁在用 | 设计要点 |
|------|-----------|--------|----------|
| `departments` | 科室 | 全系统 | v1 的数据隔离边界；员工、班次、排班周期均归属某科室 |
| `users` | **登录账号**（不是员工档案） | Web 管理员、小程序员工 | 用 `role` 区分权限：`admin` = Web 管理端登录；`staff` = 已绑定微信的小程序账号。不设独立 `admins` 表 |
| `employees` | **员工业务档案**（排班主体） | 管理员维护、排班引用 | 无论是否绑定微信都存在；被排班、绑定码、快照引用。与 `users` 分离：档案归 `employees`，登录归 `users` |
| `shift_types` | 班次类型配置 | 管理员配置 | 如白班/夜班/休息；含时间段、颜色、每日最低人数等 |
| `schedule_periods` | 某科室某周的排班周期 | 管理员编辑 | 承载草稿/已发布状态、未发布变更标记、员工端可见版本号 |
| `schedule_entries` | 管理员正在编辑的排班明细 | 管理员编辑 | 行 = 某员工在某天的班次；草稿态，员工端不可直接读 |
| `schedule_publish_snapshots` | 某次发布的班表只读副本 | 员工端只读 | 发布时生成；员工端只读最新版本，不受后续草稿修改影响 |
| `employee_binding_codes` | 员工绑定小程序的一次性凭证 | 管理员生成、员工消费 | 哈希存储；绑定成功后作废 |
| `schedule_change_logs` | 排班相关操作审计 | 管理员查看 | 记录创建、修改、发布等动作 |
| `swap_requests` | 换班申请 | — | v1.5 预留，v1 不实现 |

**`users` 与 `employees` 为何分两张表**

- `employees`：业务实体。管理员先建员工档案，再排班；员工未绑微信时也应能出现在排班表里。
- `users`：系统账号。只有需要登录的人才有记录——管理员用手机号+密码，员工绑定微信后才有 `staff` 账号。
- 不做「管理员表 + 员工表」合并成一张大表：登录字段（密码、openid）与业务字段（工号、岗位、科室）生命周期不同，拆开更清晰。

**角色与端的分工**

| 身份 | `users.role` | 登录方式 | 能做什么 |
|------|--------------|----------|----------|
| 科室管理员（如护士长） | `admin` | Web：手机号 + 密码 | 员工管理、排班、发布；**管理能力仅在 Web** |
| 普通员工 | `staff` | 小程序：微信 + 绑定码 | 查看已发布个人班表 |
| 护士长若也参与排班 | Web 使用 `admin` 账号；小程序另建 `staff` 账号并关联其员工档案 | Web 管理 + 小程序看本人班表 | 管理能力仅在 Web；小程序只按员工身份读取本人已发布班表 |

`staff` 即「员工端用户」之意，不是岗位名称。v1 中一条 `users` 记录只表示一种登录身份：`admin` 用于 Web 管理端，`staff` 用于小程序员工端。护士长通常拥有一个 `admin` 登录账号；若她也要在小程序看自己的班，需同时在 `employees` 有员工档案，并通过微信绑定流程创建一条独立的 `staff` 用户记录关联该员工档案。v1 小程序不提供管理功能。

### 7.2 核心表

#### departments - 科室

**表职责**：科室主数据，作为 v1 多科室扩展时的租户边界。



| 字段         | 类型       | 说明   | 示例 |
| ---------- | -------- | ---- | ---- |
| id         | bigint unsigned | 主键，自增 | `1` |
| name       | string   | 科室名称 | `心内科一病区` |
| created_at | datetime |      | `2026-06-20 09:00:00` |
| updated_at | datetime |      | `2026-06-20 09:00:00` |


#### users - **登录账号**（不是员工档案）

**表职责**：系统登录账号。`admin` 管 Web，`staff` 管小程序；不存放工号、岗位等员工档案信息。

| 字段            | 类型       | 说明                 | 示例 |
| ------------- | -------- | ------------------ | ---- |
| id            | bigint unsigned | 主键，自增            | `1` |
| phone         | string   | 管理员手机号，可空（staff 无） | `13800138000` |
| password_hash | string   | 管理员密码，可空           | `$2a$10$...` |
| role          | enum     | admin / staff      | `admin` |
| employee_id   | bigint unsigned | 外键，staff 关联员工      | `null` |
| wx_openid     | string   | 小程序 openid         | `oWx1234567890` |
| status        | enum     | active / disabled  | `active` |
| token_valid_after | datetime | 早于该时间签发的 Token / Web JWT 会话无效 | `2026-06-20 09:00:00` |
| created_at    | datetime |                    | `2026-06-20 09:00:00` |
| updated_at    | datetime |                    | `2026-06-20 09:00:00` |

**约束说明**

- `users.phone`：管理员账号手机号唯一；staff 用户可为空
- `users.wx_openid`：唯一，可空；一个微信只能绑定一个员工
- `users.employee_id`：staff 用户唯一，可空；一个员工只能关联一个 staff 用户
- `users.role = admin` 时，`phone` 与 `password_hash` 必填，`employee_id` 与 `wx_openid` 为空
- `users.role = staff` 时，`employee_id` 与 `wx_openid` 必填，`phone` 与 `password_hash` 为空
- `users.token_valid_after`：用于强制作废旧 Token / Web JWT 会话；停用、解绑、重新绑定、管理员改密时更新为当前时间

**字段分工**

- `users.phone` 仅用于管理员登录手机号
- `employees.phone` 用于员工档案与小程序绑定二次校验


#### employees - **员工业务档案**（排班主体）

**表职责**：员工业务档案，排班与绑定的业务主体。与 `users` 一对一（仅 `staff` 绑定后），但独立于登录账号存在。

| name          | string   |                   | `张三` |
| title         | string   | 岗位，可空             | `护士` |
| phone         | string   | 必填，员工绑定二次校验使用 | `13900139000` |
| status        | enum     | active / inactive | `active` |
| created_at    | datetime |                   | `2026-06-20 09:00:00` |
| updated_at    | datetime |                   | `2026-06-20 09:00:00` |

**唯一约束**：`(department_id, employee_no)`


#### shift_types - 班次类型配置

**表职责**：科室级班次类型字典。`duration_minutes` 表示该班次持续分钟数（如 480 = 8 小时），配合 `start_time` 可推算结束时间、跨天与工时统计；休息班可为空。

| color            | string   | 如 #4CAF50 | `#4CAF50` |
| min_required_count | int    | 每日最低覆盖人数，默认 0 | `3` |

| updated_at       | datetime |           | `2026-06-20 09:00:00` |

#### schedule_periods - 某科室某周的排班周期

**表职责**：某科室某一自然周（以周一为起）的排班容器，管理草稿/发布状态及员工端可见版本。

**表职责**：管理员当前编辑中的排班明细（草稿态）。一行 = 某员工在某天的一个班次。

| 字段            | 类型       | 说明  | 示例 |

#### schedule_publish_snapshots（推荐）- 某次发布的班表只读副本

**表职责**：每次发布时的班表全量快照。员工端唯一数据源；冗余姓名、班次等信息，保证历史展示不受后续配置修改影响。

| published_at  | datetime |         | `2026-06-20 10:00:00` |
| published_by  | bigint unsigned | 外键，发布人 | `1` |

**快照内容要求**

- 员工：`employee_id`、工号、姓名、岗位
- 班次：`shift_type_id`、代码、名称、开始时间、时长、颜色
- 排班：日期、班次、备注
- 统计：每天各班次已排人数、最低覆盖人数

| `shift_types` | `UNIQUE(department_id, code)` | 防止同科室班次代码重复 |
| `schedule_periods` | `UNIQUE(department_id, week_start)` | 防止同科室同周重复创建周期 |
| `schedule_entries` | `UNIQUE(period_id, employee_id, work_date)` | v1 限制一人一天一个主班次 |
| `schedule_publish_snapshots` | `UNIQUE(period_id, version)` | 防止发布版本号重复 |
| `users` | `UNIQUE(wx_openid)` | 防止同一微信绑定多个员工 |
| `users` | `UNIQUE(employee_id)` | 防止同一员工绑定多个 staff 用户 |
| `employee_binding_codes` | 同一 `employee_id` 仅允许一个 `active` 绑定码 | 防止多个有效绑定码并存 |

MySQL 实现时，如需表达“仅 active 绑定码唯一”这类条件唯一约束，可通过服务层事务 + 普通索引实现，或使用冗余列配合唯一索引实现；无论采用哪种方式，业务结果必须满足上表约束。

### 7.4 MySQL 字段类型约定

PRD 数据模型用于指导 Drizzle schema 落地，字段类型按以下约定实现：

| PRD 类型 | MySQL / Drizzle 建议 | 说明 |
|----------|----------------------|------|
| `bigint unsigned` | `bigint unsigned auto_increment` / `bigint unsigned` | 主键使用 MySQL 自增数字 ID；外键使用同类型 |
| `string` | `varchar(n)` | 按业务字段设置长度，如姓名 20、手机号 20、备注 255 |
| `enum` | MySQL `enum` 或受限 `varchar` | v1 可用 Drizzle `mysqlEnum` 表达 |
| `boolean` | `boolean` / `tinyint(1)` | MySQL 底层按 `tinyint(1)` 存储 |
| `date` | `date` | 排班日期、周起始日期 |
| `time` | `time` | 班次开始时间 |
| `datetime` | `datetime` | 创建、更新、发布等时间 |
| `json` | `json` | 发布快照、审计详情 |

**实现要求**

- v1 使用 MySQL 自增数字主键，不使用 UUID 主键。
- 所有主键 `id` 如无特殊说明，均为 `bigint unsigned auto_increment`。
- 所有 `*_id` 字段如无特殊说明，均为 `bigint unsigned` 外键。
- 所有业务表默认包含 `created_at`；可编辑主数据和排班草稿表包含 `updated_at`。
- Drizzle schema 中需显式声明主键、外键、唯一索引和常用查询索引。
- PRD 表格中的“外键”是业务关系说明；最终是否启用 MySQL 物理外键，可在技术实现阶段结合迁移和部署习惯决定，但数据库约束总表中的唯一性必须落库。

## 8. 关键流程

### 8.1 管理员首次使用

```
1. 使用初始管理员账号登录 Web
2. 确认 / 修改科室名称
3. 确认默认班次类型是否符合本科室
4. 配置各班次最低覆盖人数
5. 录入 10 名员工（工号、姓名、手机号等）
6. 为员工生成绑定码并分发
7. 创建下周排班 → 编辑 → 发布
8. 复制发布通知文案到微信群，提醒员工查看小程序
```

### 8.2 员工绑定流程

```
1. 员工打开小程序
2. 微信授权
3. 输入绑定码 + 手机号后四位
4. 服务端校验：
   - 绑定码存在、未使用、未过期
   - 手机号后四位匹配员工档案手机号
   - 该员工未被其他微信绑定
5. 绑定成功后绑定码失效
6. 进入「我的班表」
```

### 8.3 周排班发布流程

```
1. 管理员进入目标周排班页
2. （可选）复制上周生成草稿
3. 编辑各员工每日班次
4. 系统展示覆盖人数统计与规则警告（如有）
5. 点击「发布」→ 确认
6. 服务端开启数据库事务，并锁定目标 `schedule_periods` 行
7. 计算下一版本号，生成发布快照 version N，并更新 latest_published_version
8. 写入发布操作日志后提交事务
9. 生成微信群通知文案
10. 员工小程序刷新后可看到新班表与最新发布时间
```

### 8.4 已发布班表修改流程

```
1. 管理员打开已发布周期
2. 修改单元格 → 周期标记 has_unpublished_changes = true
3. 员工端仍展示上一版发布快照
4. 管理员完成修改后再次发布
5. 生成新版本发布快照，员工端更新
```

## 13. 风险与对策


| 风险          | 影响       | 对策                    |
| ----------- | -------- | --------------------- |
| 各医院班次时间差异大  | 产品不适配    | 班次类型完全可配置             |
| 大夜班跨日逻辑出错   | 展示混乱     | 统一「开始时刻 + 时长」存储，充分测试  |
| 员工不愿绑定小程序   | 推广失败     | 绑定码流程极简；管理员侧可看到绑定进度   |
| 绑定码泄露或误用     | 数据越权     | 绑定码一次性使用，配合手机号后四位校验 |
| 已发布后又改班引发纠纷 | 信任问题     | 发布快照 + 最新发布时间 + 变更日志    |
| 小程序订阅消息限制   | 触达不足     | v1 生成微信群通知文案；v1.5 做订阅 |
| 覆盖人数配置不符合实际 | 漏排或误报    | 班次最低人数可配置，发布前展示警告摘要   |
| 规则变复杂       | scope 膨胀 | 严格区分 v1 警告 vs v2 强制规则 |
